# 決定ログ

用語の読み方や分類について、自明ではない判断をしたときの記録。
新しいレコードが `pronunciation_rules.md` の優先順位3(暫定採用)に
該当する場合や、分類に迷った場合は、ここに追記すること。
古い記録も削除せず、時系列で残す。

---

## 2026-07-19: 辞書管理システムの新設と初期データの移行元

`infra/lambda/conversation/index.js` にハードコードされていた
`TTS_PRONUNCIATION_OVERRIDES`(AWS/S3/EC2/API/URL/SDK/CDK/HTTPS/HTTP/JSON/CLI
の11件と、キャラクター名取の「せんせえ」の読み替え)を、本辞書管理
システムの初期データとして移行した。

移行時の判断:

- 上記11件の技術用語は、それぞれ最も近いカテゴリ
  (`aws` / `programming` / `web` / `networking` / `dev_tools`)へ振り分けた。
  1文字ずつ読む頭字語読みは元の実装をそのまま踏襲している。
- 「せんせえ → せんせい」は技術用語ではなく、キャラクター「名取」の
  台詞に固有の読み替えであるため、`custom` カテゴリに `app_specific`
  タグを付けて登録した。将来的に他のアプリ固有の言い回しが増えた場合も
  同様に `custom` + `app_specific` に集約する方針とする。
- `CDK` は文脈上ほぼ常に「AWS CDK」を指すため、`aws` カテゴリに分類した
  (`id: aws-cdk`)。

## 2026-07-19: マスタデータ拡充(技術用語マスタ作成タスク)

ダミーデータを実用データへ置き換え、18カテゴリ中17カテゴリ(custom を除く)
に用語を追加した。判断に迷った点を記録する。

### validate.js の自己衝突バグを修正

`dictionary.js` の `validate()` が、1レコード内で `term` と `aliases` に
大文字小文字違いの表記(例: `term: Docker` / `aliases: [docker]`)を
持たせただけで「衝突」と誤検知するバグがあった。本タスクの要件
(aliasesに大文字小文字違いを含める)を満たすために、レコード自身の
表記同士はローカルにまとめてから、他レコードとのみ照合するよう
2パス方式に修正した(グローバルなseen集合への登録はチェック完了後に行う)。

### 読みが割れる語の採用理由

- **RAG**: 「アールエージー」(1文字読み)と「ラグ」(単語読み)で揺れる
  が、実際の技術会話では単語読みが広く定着しているため「ラグ」を採用。
  `ambiguous_reading` タグを付与。
- **Llama / LLaMA**: 頭字語としての由来(Large Language Model Meta AI)
  上は1文字読みもあり得るが、口語では動物名由来の「ラマ」がほぼ全面的に
  使われているためこちらを採用。`ambiguous_reading` タグを付与。
- **JWT**: 「ジェイダブリューティー」(1文字読み)と口語の「ジョット」で
  揺れる。後者は文脈依存性が高く確信が持てなかったため、より広く通用する
  1文字読みを暫定採用。`ambiguous_reading` タグを付与。
- **Postgres**: 「PostgreSQL」の正式な1文字読みとは別に、口語で定着した
  「ポスグレ」を採用。読みが異なるため `PostgreSQL` とは別レコードとして
  登録した(1レコード1読みの原則)。`ambiguous_reading` タグを付与。

### 登録を見送った語

- **CORS**: 実務での読み方(「コルス」「シーオーアールエス」等)の確信が
  持てなかったため、推測で登録することを避け見送った。
- 一般的な英単語と衝突しやすい語は、より安全な複合表記のみを登録し、
  裸の単語(`Spring`、`Rails`、`Express`、`Go` 単体等)は term にも
  alias にも含めていない(例: `Spring` → `Spring Boot` のみ登録)。

### カテゴリのスコープについて

- `statistics`・`japanese_technical` は、実際の音声会話での登場頻度に
  ついて確信が持てる語のみに絞ったため、他カテゴリと比べて件数が
  少ない。特に `japanese_technical` は他カテゴリ(ローマ字表記の
  読み間違い防止)とは異なり、漢字の読みが割れやすい和製技術用語
  (冪等性・閾値等)を対象とした。実際の利用状況を見て今後拡充が
  必要か判断すること。

### 今後の課題(index.js 連携時の注意点)【2026-07-19 接続完了・解消済み】

(下記の懸念は "2026-07-19: index.js への接続" エントリで対応済み。
記録として残す。)

`infra/lambda/conversation/index.js` の `toTtsText()` は、`term` が
`/^[A-Za-z0-9]+$/`(純粋な英数字)に一致しない場合、正規表現の特殊文字を
エスケープせずにそのまま `RegExp` へ渡す実装になっている。本ラウンドで
登録した `.NET`・`C#`・`Node.js`・`React.js`・`CI/CD`・`TCP/IP` 等は
`.`・`#`・`/` のような正規表現の特殊文字(または偶然一致しない文字)を
含むため、将来 `generated/lookup.json` を `toTtsText()` に接続する際は、
先に `term`/`alias` を `RegExp` 用にエスケープする処理を追加する必要が
ある。現状は未接続のため実害はないが、接続時に必ず対応すること。

## 2026-07-19: 発音表記ルールの詳細化と既存データの読み修正

外部から提示された詳細な発音表記ルール案をもとに `taxonomy/pronunciation_rules.md`
を全面的に書き直し、既存レコードのうち規約と食い違っていたものを修正した。

### 読みを修正した既存レコード

- **IAM**: `アイエーエム`(1文字読み)→ `アイアム`(単語読み)。実際の
  AWS技術者コミュニティでは "I am" のように単語として読まれるのが
  通例のため修正。
- **TensorFlow**: `テンサーフロー` → `テンソルフロー`。機械学習分野で
  すでに定着している「テンソル」という訳語に合わせた表記へ修正。
- **MongoDB**: `モンゴディービー` → `モンゴーディービー`(長音追加)。
- **JWT**: `ambiguous_reading` タグを削除。「ジェイダブリューティー」が
  国内技術者コミュニティで広く通用する読みであることが確認できたため、
  暫定採用から確定へ変更した。

### alias とベンダー正式名称の読み一致について

`term: EC2` / `aliases: ["Amazon Elastic Compute Cloud"]` のように、
ベンダー名を伴う正式名称を略称と同じ `reading` で alias 登録する運用は、
初版データの `AWS`(`aliases: ["Amazon Web Services"]` → `エーダブリューエス`)
の時点からすでに採用されていた設計であることを確認した。実際の会話では
`AWS`/`Amazon` 等の前置き部分が読み上げられず略称のみ発音されるのが通例
であるため、この運用を意図的な設計として `pronunciation_rules.md` §11に
明文化した(1レコード1読みの原則そのものは維持しつつ、「略称の読みを
正式名称にも割り当てる」ケースを許容する形で整理)。一方、`JS`(`JavaScript`
の略で読みが異なる)のように略称自体の読みが展開形と異なる場合は、
引き続き別レコードとして登録する。

### reading にスペースを含めない方針の再確認

提示された規約案には `IAM Identity Center → アイアム アイデンティティセンター`
のように読みの区切りにスペースを含む例が含まれていたが、既存の
`READING_PATTERN`(平仮名・片仮名・長音記号のみ)はスペースを許容しない。
この制約を変更する理由はないと判断し、該当する新規レコード(IAM Identity
Center、HTTP/2、HTTP/3、GPT-4.1 等)はスペースを除いた1つの連続文字列
として登録した。

## 2026-07-19: index.js への接続

`generated/lookup.json` を `infra/lambda/conversation/index.js` の
`toTtsText()` に接続した。旧来のハードコード `TTS_PRONUNCIATION_OVERRIDES`
(12件、本辞書へ移行済みだったもの)は削除した。

### 接続方式

`infra/lambda/conversation` は `Code.fromAsset('lambda/conversation')` で
ディレクトリをそのままzipするだけで、esbuild等のバンドラーを介さない
(`infra/lib/infra-stack.ts` 参照)。そのため `generated/lookup.json`
(git管理外のビルド成果物)をLambdaへ持ち込む手段として、`build.js` に
`infra/lambda/conversation/pronunciation-lookup.json` への書き出しを
追加し、このファイル自体をgit管理下に置いた(`system-prompt.md` が
同ディレクトリに直接コミットされているのと同じパターン)。デプロイ時に
ビルドを自動実行する仕組みは無いため、辞書を更新したら
`npm run build` → 差分コミット → `cdk deploy` を人間が順に実行する
運用とする(README.md「更新フロー」参照)。

### toTtsText() の書き直し

以前から記録していた「非ASCII termを正規表現の特殊文字としてエスケープ
せずに使っている」問題(本ログの「今後の課題」参照)をこの接続を機に
解消した。

- 全termを `RegExp` 用にエスケープしてから `RegExp` を組み立てる。
- 境界判定を `\b`(ASCII専用)から、直前直後がASCII英数字でないことを
  要求する先読み・後読み `(?<![A-Za-z0-9])...(?![A-Za-z0-9])` に統一。
  日本語termにも一律適用でき、`\b` が効かない問題も解消される。
  `AI` が `AIRPORT` の内部で誤爆しないことをテストで確認した。
- term を文字数の降順でソートしてからマッチングし、`CI/CD` のような
  複合語がその部分文字列(`CI`・`CD`)による先行置換で壊れないように
  した。

### 大文字小文字の扱い

`toTtsText()` の置換は大文字小文字を区別する(`g` フラグのみ、`i` は
付けない)。`validate()` の重複チェックは大文字小文字を無視するが、
これは「表記ゆれを検出する」ためであり、「実際の置換を大文字小文字
無視にする」こととは別の話である。`Docker`/`docker` のような表記ゆれは
(マスタ拡充タスクで意図した通り)aliasとして個別に登録し、置換自体は
登録された表記と完全一致した場合のみ行う設計を維持した。

## 2026-07-24: `sensei-address`(せんせえ)の読みをアクセント問題で再修正

**症状**(ユーザー報告、実機TTS聴取): `せんせえ` → `せんせい` という
既存の変換で、ElevenLabsの読み上げが「先生」(teacher)のアクセントでは
なく、まったく別語である「慣性」に近いイントネーションになっていた。
`reading` は平仮名・片仮名によるかな読みのみでアクセント位置・高低を
保持しない(`pronunciation_rules.md` 冒頭)という制約上、同じかな文字列
`せんせい` に対してTTSエンジンが選ぶアクセントパターンを辞書側から直接
制御することはできない。

**対応**: `reading` を `せんせい` → `せんせー`(長音記号)に変更した。
これは `pronunciation_rules.md` §1 が例示する一般原則(「和語・漢語由来の
読みでは長音記号ではなくあ/い/う/え/お行の文字を使う。例:「せんせい」で
あって「せんせー」ではない」)への**意図的な例外**である。この語はキャラ
クター「名取」固有の呼びかけ表現であり一般的な和語ルールの対象外である
こと、かつ実際にElevenLabsで聴き比べた結果、長音記号表記の方が「先生」
本来のアクセントに近い読み上げになったことから、個別レコードの例外として
採用した(`pronunciation_rules.md`「保守方針」: 例外は個別レコードで管理し、
ルール自体は変更しない)。`ambiguous_reading` タグを付与した。

一般的な和語における長音記号不使用の原則(`pronunciation_rules.md` §1)は
変更していない。この例外はTTSエンジンのアクセント推定の癖に対する
ピンポイントの回避策であり、他の語に無条件で適用すべきパターンではない。
