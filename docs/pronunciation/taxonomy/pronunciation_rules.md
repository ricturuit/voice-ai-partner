# 発音表記ルール

`reading` フィールドの書き方を定める。特定のTTSサービスに依存しない
表記(平仮名・片仮名によるかな読み)を正としており、各TTSサービスへの
出力方法は `generated/` を消費する側(将来のアダプタ層)の責務とする。

対象は音声会話AI「名取」で利用する技術用語であり、特定のTTSエンジンには
依存しない。`reading` はアクセント位置・高低・韻律を保持しない
(将来の発音レイヤー拡張方針については `README.md` を参照)。

## 基本方針

- `reading` は自然な日本語読みとする。
- 技術者が日常会話で読む発音を優先する(ベンダー公式表記より優先)。
- カタカナ(または平仮名)で統一する。
- アクセント記号・IPA・長音記号以外の補助表記は使用しない。
- 説明文・用途・品詞は含めない。
- 正式名称ではなく、一般的な呼び方を採用する。
- 読みが複数存在する場合は国内で最も一般的な読みを採用する(「優先順位」参照)。

## 1. 文字種

- `reading` は **平仮名・片仮名・長音記号(ー)のみ**で書く。
  英数字・漢字・記号(・など)・スペースは含めない。
- 促音(っ)・拗音(ゃゅょ)・撥音(ん)は通常の日本語表記どおりに使ってよい。
- 長音は原則として片仮名語では「ー」、和語・漢語由来の読みでは
  「あ/い/う/え/お」行の文字を使う(例: 「せんせい」であって「せんせー」ではない)。
- 複数語からなる用語(例: `IAM Identity Center`)でも、`reading` は
  スペースなしで1つの連続した文字列として書く
  (例: `アイアムアイデンティティセンター`)。単語の区切りを示したい
  場合でも `reading` にスペースは入れない。

`validate` はこの文字種チェックを機械的に行う(正規表現で
`^[ぁ-んァ-ヶー]+$` に一致するかを検証する)。

## 2. アルファベット略語の読み方

- 業界的に「1文字ずつ読む」ことが定着している略語は、文字ごとの読みを
  連結する(例: `AWS` → `エーダブリューエス`、`API` → `エーピーアイ`)。
- 一般的に「単語として読む」ことが定着している略語は、その読みをそのまま
  当てる(例: `IAM` → `アイアム`、`SAM` → `サム`)。
- どちらか判断が難しい語は `ambiguous_reading` タグを付け、
  `decision_log.md` に採用理由を残す。

## 3. 数字・単位

- 数字を含む語(例: `EC2`、`S3`)は、数字部分も含めて自然な読みを
  当てる(`EC2` → `イーシーツー`、`S3` → `エススリー`)。
- バージョン番号やIPアドレス系の語も同様に、数字を自然な読みに変換して
  1つの連続文字列にする(`IPv4` → `アイピーブイフォー`、
  `HTTP/2` → `エイチティーティーピーツー`、`GPT-4` → `ジーピーティーフォー`、
  `GPT-4.1` → `ジーピーティーフォーポイントワン`)。表記上「.」や「/」で
  区切られていても、reading側では区切り記号もスペースも入れない。

## 9. 記号を含む用語

記号は日本で一般的な読みへ変換する。`term`/`alias` 自体には記号を
残してよい(正規表現の特殊文字を含む場合の実装上の注意は
`decision_log.md` を参照)。

## 10. カタカナ語(定着済みの外来語)

すでに日本語として定着しているカタカナ語は、一般的なカタカナ表記を
そのまま `reading` に使う。無理に英語発音へ寄せない。

## 11. 別表記(alias)の扱い

- `aliases` は `term` と**同じ `reading`** を持つ別表記の一覧である。
  読みが異なる場合は別レコードとして登録する(1レコード1読みの原則)。
- ベンダー名を伴う正式名称(例: `AWS CDK`、`Amazon Elastic Compute Cloud`)
  は、実際の会話では略称のみが発音され、`AWS`/`Amazon` 等の前置き部分は
  読み上げられないのが通例である。そのため `term` の略称と同じ
  `reading` を割り当てて `aliases` に含めてよい
  (例: `term: EC2`、`aliases: ["Amazon Elastic Compute Cloud"]` は両方とも
  `イーシーツー` と読ませる)。
- 一方、`JS`(`JavaScript` の略)のように**略称自体の読み方が展開形と
  異なる**場合は、alias にせず別レコードとして登録する。

## reading に含めないもの

以下は `reading` に記載しない。

- 説明文
- 括弧
- 英語表記
- 発音記号・IPA
- アクセント記号
- 用途
- 品詞

## 優先順位

読みが複数存在する場合は以下を優先する。

1. 日本国内の技術者コミュニティで最も一般的な読み
2. ベンダー公式の発音
3. 英語の原音
4. その他の読み

上記で判断が難しい場合は、最も自然で誤解の少ない読み方を暫定採用し、
`ambiguous_reading` タグを付けて `decision_log.md` に理由を残す。

## 保守方針

- 新しい用語も本ルールに従って登録する。
- 既存レコードも本ルールに合わせて修正する。
- 同じ読み方のルールを全カテゴリで統一する。
- 例外が必要な場合は、個別レコードで管理し、ルール自体はできるだけ
  シンプルに保つ。
