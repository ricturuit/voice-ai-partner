実機iOS/Androidビルドは後回し。標準の `flutter create` 構成のままなので、

## 修正: 長い返答で音声が途中で止まる(2026-07-23、v1.0.1)

**症状**(ユーザー報告): v1.0.0の再設計以降、文章が長くなると返答の音声が
途中で止まってしまうことがあった。それ以外の会話フロー(自動再生・手動
再生・ミュートスイッチ追従など)は改善されたとの評価。

**原因**: `AudioPlaybackService.play()`の完了待ちに設けていた
`_completionTimeout`(30秒)が、実際に鳴っている途中の音声を強制停止
(`element.pause()`)していた。この30秒は「`<audio>`要素が本当にハング
した場合の保険」のつもりで入れた値だったが、日本語の音声読み上げは
数百文字程度でも普通に30秒を超えることがあり、正常な長さの返答まで
巻き込んで打ち切ってしまっていた。バックエンド(`CLAUDE_MAX_TOKENS=4096`)
側の直近のCloudWatchログを確認したところ、テキスト生成自体の切り詰め
(`stop_reason === "max_tokens"`)は発生しておらず、原因はこのクライアント
側タイムアウトのみと確認できた。

**対応**: `_completionTimeout`を30秒→5分に引き上げた。あくまで
「本当にハングした場合の保険」であり続けるべきという位置づけは変えず、
現実的などんな1ターンの返答よりも十分長く、かつ無期限にハングすることは
防げる値として設定した。返答テキストの長さ自体に上限を設ける他の箇所
(`CLAUDE_MAX_TOKENS`)には手を入れていない(切り詰めが発生していない
ため、現状維持で問題ないと判断)。

**確認**: `flutter analyze`/`flutter test --platform chrome`は問題なし。
本番へデプロイ済み。**実機で、長めの返答でも音声が最後まで再生される
ようになったか確認をお願いしたい**。

## 画面構成

- テキスト入力欄 + マイクボタン + 読み上げ強制停止ボタン + 送信ボタン(下部)
- マイクボタンをタップすると「ポン」という合図音が鳴り、音声入力(STT)を
  開始。認識結果はテキスト入力欄にリアルタイムで反映される。**送信は
  常に明示的な操作でのみ行われる**(無音検知による自動送信タイマーは
  2026-07-22に廃止 — 下記「修正」参照): もう一度マイクボタンをタップする
  (`stopListeningAndSend()`)か、送信ボタンを押す
- 送信〜AIの返答の読み上げが完了するまでの間は、**マイク入力と送信ボタンが
  ロックされる**(テキスト入力欄自体には引き続き入力できるが、送信はできない)。
  読み上げが完了すると再び「ポン」と鳴り、マイク入力・送信ボタンが使える
  ようになる。次の発話をするにはもう一度マイクボタンをタップする(自動では
  再開しない — 理由は下記の実装メモを参照)
- 読み上げ中に「読み上げを停止」ボタンを押すと、その場で読み上げを打ち切り、
  すぐにマイク入力・送信ボタンが使えるようになる
- 送信ボタンは常に表示されたまま。テキストを直接入力して送信ボタンを押す
  従来の経路も引き続き利用可能(音声・テキストどちらからでも会話できる)
- 会話履歴を吹き出し表示(ユーザー発言は右・青、AI返答は左・グレー、エラーは赤)
- AI返答の音声は自動再生。ブラウザの自動再生ポリシーでブロックされた場合に備え、
  吹き出し内に「音声を再生」ボタンで手動再生も可能

## 音声入力(STT)の実装

- パッケージ: [`speech_to_text`](https://pub.dev/packages/speech_to_text)
  (Web版は`webkitSpeechRecognition`/`SpeechRecognition`(Web Speech API)を
  利用する実装が組み込み済み。追加パッケージ不要)
- アプリ起動時に`SpeechToText.initialize()`でブラウザの対応状況を確認し、
  対応していなければマイクボタンをグレーアウト(`Icons.mic_off`)表示
- マイクボタンタップ時にも改めて利用可否をチェックし、非対応・エラー時は
  SnackBarで理由を表示してテキスト入力にフォールバックできるようにしている
  (`_handleSpeechError` / `_toggleListening`、`chat_screen.dart`)
- `initialize()`・`listen()`双方をtry/catchで保護し、ブラウザ側の予期しない
  例外でアプリ全体がクラッシュしないようにしている

### 無音検知による自動送信は廃止済み(2026-07-22)

`speech_to_text`のWeb実装は`continuous: true`でSpeechRecognitionを開始する
(`partialResults`と連動)ため、**ブラウザ側は途中でどれだけ間が空いても
「発話終了」を自動検知してくれない**(`result.finalResult`が自然にtrueに
なることはなく、`.stop()`を呼んだ時に初めて最後の結果がfinalとして届く)。

以前はこれをアプリ側で無音タイマー(`Timer.periodic`)により独自に検知し、
一定秒数(2〜10秒、AppBarのメニューで変更可)無音が続いたら自動送信して
いたが、**この仕組み自体を完全に廃止した**。理由は「修正: 無音自動送信
タイマーを廃止し、明示タップのみの送信に統一」の項を参照 — 自動送信は
ユーザーの実際のタップ操作を経ずに`sendText()`を呼ぶため、返答の自動再生に
必要な「ユーザー操作起点」の条件を満たせず、応答が遅い("長考")ほど
自動再生が失敗しやすいという構造的な問題があった。

### ターン制の会話フロー(自動での聞き取り再開はしない)

送信〜返答の読み上げ完了までは`_isSending` / `_isPlayingReply`で
マイクボタンと送信ボタンをロックし(テキスト入力欄自体は常に有効)、
読み上げ完了(または「読み上げを停止」ボタンでの強制終了)のタイミングで
両方のボタンを解放する。**読み上げ完了後にマイクの聞き取りを自動的に
再開することはしない** — 当初は「無音区間経過→送信→そのまま次の発話を
自動で聞き取り再開」というハンズフリーループを実装していたが、実機検証で
以下の問題が解消しきれなかったため、「毎ターン、ユーザーがマイクボタンを
押す」というシンプルな設計に変更した:

- ブラウザが`.stop()`直後の`.start()`を拒否し、返答待ち中に
  「音声入力を開始できませんでした」というエラーで聞き取りが止まってしまう
  ことがあった(リトライ処理を入れても解消しきれなかった)
- `AudioPlayer.play()`が返す`Future`は**再生が「開始」した時点で完了して
  おり、読み上げが終わるまでは待ってくれない**という`audioplayers`の
  挙動により、読み上げ中に聞き取りが再開されて無音カウントが誤発火する
  問題があった。`AudioPlayer.onPlayerComplete`ストリームを購読して実際の
  完了イベントを待つよう修正はしたが、自動再開の仕組み自体をやめた方が
  シンプルかつ確実と判断した

### 合図音(「ポン」)

控えめな「ポン」音(`assets/sounds/silence_cue.wav`、音量0.18、専用の
`AudioPlayer`インスタンスで再生)を、**「今話しかけてよい」タイミングで**
鳴らす:

- マイクボタンをタップして聞き取りが実際に始まった瞬間(初回・毎ターンとも)
- AIの返答の読み上げが完了した瞬間(「読み上げを停止」ボタンで強制終了した
  場合も含む)

既存の著作権付き効果音(iOS標準の通知音など)は使わず、短い正弦波トーン
(約220ms、880Hz、指数減衰エンベロープ)をこちらで新規に合成したものを
同梱している。

### 解決済み: マイク入力にはHTTPSが必須だった件

**Web Speech API(および`getUserMedia`によるマイクアクセス全般)は、ブラウザの
「セキュアコンテキスト」(HTTPSまたは`localhost`)でのみ動作する。** S3の
静的website hosting単体はプレーンHTTPしか提供できないため、そのままでは
マイク入力が動作しなかった。

実際に検証して切り分けた内容:
- `https://`または`http://localhost`(secure context)では、
  `SpeechRecognition`が正常に初期化され、マイクボタンが有効になる
- プレーンHTTP(非localhost)では `window.isSecureContext` が`false`になり、
  `navigator.mediaDevices` 自体が存在しなくなる。`SpeechRecognition`コンス
  トラクタ自体は存在するためマイクボタンは一見有効に見えるが、タップして
  実際に音声認識を開始しようとすると失敗する

**対応**: S3バケットの手前にCloudFrontディストリビューションを追加し(`infra/lib/infra-web-stack.ts`)、
デフォルト証明書(`*.cloudfront.net`、独自ドメイン・ACM証明書は不要)でHTTPS配信するようにした。
`cloudfront:*`権限が不足していたため、`claude-code-dev`ユーザーにインラインポリシーで追加。
IAM管理ポリシーの上限(10個)に達していたため、追加の管理ポリシーではなくインラインポリシー
(`CloudFrontAccess`)で対応した。

検証フェーズ限定の暫定構成という位置づけのため、価格クラスは最小(`PRICE_CLASS_100`)、
WAF等の追加設定はしていない。ネイティブモバイルアプリへの移行後にこのWeb版が不要になれば、
`InfraWebStack`ごと`cdk destroy`すれば消せる。

## 修正済み: MissingPluginException(音声入力が`initialize`で失敗)(2026-07-13)

**症状**: 音声入力(マイクボタン)を使おうとすると
`MissingPluginException(No implementation found for method initialize on
channel plugin.csdcorp.com/speech_to_text)` が発生していた。

**原因**: `.dart_tool/flutter_build/` 配下のインクリメンタルビルドキャッシュが
不整合を起こしていた。`web_plugin_registrant.dart`(Web版プラグイン登録用の
自動生成ファイル)のソース自体は`SpeechToTextPlugin.registerWith(registrar)`
を正しく含んでいたが、実際にビルドされた`main.dart.js`には
`webkitSpeechRecognition`関連のコードが含まれておらず、代わりにデフォルトの
MethodChannel実装(ネイティブアプリ向けの、存在しないチャンネルにメッセージを
送ろうとする実装)だけが残っていた。`speech_to_text`パッケージを追加した後、
一度も`flutter clean`せずに複数回ビルドを重ねたことで、古いビルド成果物が
一部再利用されてしまったことが原因と考えられる。

**対応**: `flutter clean` → `flutter pub get` → クリーンな状態で
`flutter build web --release --no-web-resources-cdn --dart-define=...` を
再実行。ビルド後の`main.dart.js`に`webkitSpeechRecognition`
(2箇所)・`SpeechRecognitionError`・`SpeechRecognitionResult` が含まれる
ことを確認してからデプロイ・CloudFrontキャッシュ無効化を実施。

`web/index.html`(`flutter_bootstrap.js`の読み込み等)は元々標準のまま
正しく、修正不要だった。

## 修正済み: 送信後ロード表示が終わらず入力欄が固まる不具合(2026-07-13)

**症状**(実機報告): テキストを送信すると返答テキストは表示されるが、
音声が再生されないまま送信中のロード表示が終わらず、以降テキスト入力欄が
無効化されたまま操作できなくなる。

**原因**: `_handleSend()`内で、返答音声の自動再生(`_playAudio(...)`の
`await`)が`_isSending`を`false`に戻す`finally`ブロックより**前**、かつ
同じ`try`ブロック内にあった。一部のブラウザでは自動再生がブロックされた際、
`audioplayers`が使う`<audio>`要素の`play()`(内部的にWeb Audio APIの
`AudioContext.resume()`を経由)が**例外を投げずに永遠に保留されたまま
(pending)になる**ケースがあり、この場合`await`が完了しないため
`finally`にも到達できず、`_isSending`が`true`のまま固定される。
`TextField`・送信ボタンとも`enabled: !_isSending`相当の条件で無効化して
いたため、結果的に入力できなくなっていた。

**対応**:
1. 音声の自動再生を`_isSending`の管理対象から完全に分離。API応答を受け
   取った時点(`finally`)で`_isSending`を`false`に戻し、**その後で**
   音声再生を試みるように順序を変更(送信・入力の可否が音声再生の成否に
   左右されなくなった)
2. `_playAudio()`に`.timeout(Duration(seconds: 10))`を追加。ブラウザ側の
   挙動に関わらず、再生試行が10秒以内に必ず完了(失敗含む)するように
   した(手動の「音声を再生」ボタンでも同様に保護される)

## 修正済み: 音声が再生されない不具合(2026-07-13)

**症状**: テキスト返答は表示されるが、自動再生・手動再生ボタンのどちらでも
音声が再生されなかった。

**原因**: 音声保存用S3バケット(`voice-ai-partner-artifacts-...`)にCORS設定が
存在しなかった。`audioplayers` のWeb実装(`audioplayers_web`)は、音量・
パン(左右バランス)調整のためWeb Audio APIの`MediaElementAudioSourceNode`に
`<audio>`要素を接続する構造になっており、そのために`<audio>`要素へ
`crossOrigin = "anonymous"` を明示的に設定している
(`wrapped_player.dart`)。この設定がある`<audio>`要素はブラウザが
CORSチェック付きリクエストとして音声を取得しようとするため、S3側に
`Access-Control-Allow-Origin` 等が無いと読み込みに失敗し、再生されない
(かつ`chat_screen.dart`側で例外を握りつぶしていたためエラーも見えなかった)。

**対応**:
1. `infra/lib/infra-core-stack.ts` の `ArtifactsBucket` にCORS設定を追加
   (`GET`/`HEAD`許可、`AllowedOrigins: ["*"]`)。バケット自体は署名付きURL
   経由でしか読めない非公開設定のままなので、CORSを緩めても認可の範囲は
   広がらない(ブラウザJSが結果を読めるかどうかだけの話)
2. `chat_screen.dart`: 再生失敗時に必ず `debugPrint` でログを残すよう変更。
   自動再生の失敗(ブラウザの自動再生ポリシーでブロックされるケース)は
   従来通り無言だが、「音声を再生」ボタンでの手動再生が失敗した場合は
   SnackBarでエラーを表示するようにした(今後同種の問題が起きた際に
   気づけるように)

**検証**: 修正後、S3にCORS設定が反映されたことを`get-bucket-cors`で確認。
実際に新しい音声ファイルを生成し、ブラウザの`<audio crossorigin="anonymous">`
が送るのと同じ形のリクエスト(`Origin`ヘッダー付き、`Range`リクエスト込み)を
curlで再現したところ、`Access-Control-Allow-Origin: *` 等が正しく返り、
`206 Partial Content` でオーディオデータを取得できることを確認した。
ヘッドレスブラウザでの実再生確認は、このセッションのサンドボックス環境が
S3ドメインへの直接アクセスを許可していない制約により完走できなかった
(同じ制約は以前の動作確認時にも発生している。実機のブラウザは通常の
インターネット接続を使うため、この制約の影響を受けない)。

## 既知の制限: 日本語フォントがGoogle Fonts CDNに依存している

`--no-web-resources-cdn` はCanvasKit/Roboto本体には効くが、Flutter Webは
日本語などCJK文字のグリフを表示する際、Noto Sans SCなどのフォールバック
フォントを実行時に `fonts.gstatic.com` から動的取得する仕組みが別途動いている
(この会話アプリの主目的が日本語表示なので影響が大きい)。通常のインター
ネット環境なら問題なく取得できるが、Google Fonts CDNをブロックする
ネットワーク(一部の企業網、中国本土など)ではテキストが表示されない。

恒久対応するなら、日本語フォント(Noto Sans JPなど)をアプリのassetとして
同梱し `ThemeData(fontFamily: ...)` で明示指定して動的取得自体を発生させない
のが良い。Phase 1では未対応(現状のフォールバック取得で通常は動く)。

| 状態 | 見た目 | ラベル | タップ時の動作 |
|---|---|---|---|
| 待機中 (idle) | 通常色・アイコン`mic_none` | タップして話す | `startListening()` (STT開始) |
| 聞き取り中 (listening) | 赤色・パルスアニメーション・`mic` | タップでキャンセル | `cancelListening()` (何も送信せず待機中に戻る) |
| 送信中 (sending) | グレー・`hourglass_top`・無効化 | 考え中… | 無効化されているためタップ不可 |
| 読み上げ中 (playingReply) | 青色・`volume_up` | タップで停止 | `forceStopReading()` (即座に強制終了し待機中に戻る) |

読み上げ完了(`onPlayerComplete`)時は既存のチャットモードと同じ経路で
自動的に待機中へ戻る。無音判定による自動送信、既存の「ポン」cue(聞き取り
開始時・読み上げ完了時)もチャットモードと全く同じロジックをそのまま流用
している(`ConversationController` 経由で共有しているため)。

新規メソッド `cancelListening()` は、既存の `stopListeningAndSend()`
(チャットモードのマイクボタン: 聞き取り中に再タップすると認識済みテキストを
送信する)とは意図的に区別した — 音声会話モードでの「タップでキャンセル」は
何も送信せずに待機中へ戻る必要があるため。

### 自動再生アンロックへの流用

新しく追加した無音(全サンプル0)の短いWAV(`assets/sounds/unlock_silent.wav`,
100ms)を、`startListening()` の最初の行で(どの `await` より前に、同期的に)
`audioPlayer.play(..., volume: 0.0)` として再生している。これにより、
音声会話モードの円形ボタンをタップする行為(待機中→聞き取り中への遷移)が
実際のユーザー操作(タップ)と同じ呼び出しスタック内で発生する`play()`と
なり、ブラウザの自動再生ポリシー上「ユーザー操作起点の再生」として扱われる。
これによって、後で(ユーザー操作から切り離れたタイミングで)発生する
AI応答の音声の`play()`が同じ`AudioPlayer`/`<audio>`要素経由でブロックされ
にくくなることを狙っている。チャットモードのマイクボタンも同じ
`startListening()` を経由するため、両モードで同様にアンロックが働く。

## 修正: 音声会話モードのタップ/長押し操作とテキスト送信時の自動読み上げ(2026-07-13)

### 音声会話モード: ワンタップ=入力終了、長押し=キャンセル

**症状**: 音声会話モードで周囲の音(生活音など)を拾い続けてしまい、無音
判定が成立しないため「聞き取り中」のまま終了しなくなっていた。

**対応**: 聞き取り中のボタン操作を分離した。
- **ワンタップ**: `stopListeningAndSend()` を呼び、その時点で認識できて
  いるテキストで入力を終了・送信する(ラベル: 「タップで入力終了」)
- **長押し**: `cancelListening()` を呼び、何も送信せず待機中に戻る
  (ラベル: 「長押しでキャンセル」を小さく併記)

無音判定タイマー自体は残しているが(静かな環境なら従来通り自動送信される)、
周囲の音で無音判定が成立しない環境でも、ユーザーが手動でワンタップして
入力を締め切れるようにした。待機中・送信中・読み上げ中の各状態や、既存の
`cancelListening()`/`stopListeningAndSend()` メソッド自体には変更はない
(呼び出し元をボタンの単押し/長押しで入れ替えただけ)。

### テキスト入力時に回答が自動で読み上げされない不具合

**症状**: 音声入力(マイクボタン)を一度も使わず、テキスト入力→送信のみで
やり取りすると、返答は表示されるが音声が自動再生されなかった。

**原因**: 前回追加した音声アンロック処理(`_unlockAudioPlayback()`)は
`startListening()` の中でしか呼んでいなかった。ブラウザの自動再生ポリシー
は、あるユーザー操作(クリック/タップ/Enterキー)から一定時間内
(「transient activation」と呼ばれる、数秒程度)に発生した`play()`のみを
許可する。Claude+ElevenLabsのAPI往復には数秒かかることがあり、送信ボタン
クリックからその応答が返ってくるまでの間にこの許可時間が切れてしまうため、
返答音声の`play()`が(ユーザー操作から離れたタイミングでの呼び出しと
みなされ)ブロックされていた。一方、マイクボタン経由の場合は
`startListening()`内で同じ`AudioPlayer`インスタンスに対して無音再生を
一度実行しており、`audioplayers`が使う`<audio>`要素は一度でも成功した
再生があれば以降はユーザー操作なしでも再生できるようになる、という
ブラウザの挙動を利用してアンロックできていたが、マイクを一度も使わない
テキストのみのやり取りではこの「最初の1回」が発生していなかった。

**対応**: `ConversationController.sendText()` の冒頭(バリデーション直後、
最初の`await`より前)でも同じ`_unlockAudioPlayback()`を呼ぶようにした。
送信ボタンのクリックハンドラ・テキストフィールドのEnterキー送信のどちらも
`sendText()`を直接同期呼び出ししているため、この無音再生がクリック/
Enterキーと同じ呼び出しスタック内で実行され、以降のAPI応答待ちが長引いても
返答音声の自動再生がブロックされにくくなる。

**確認**: `flutter analyze`/`flutter test`は問題なし。実際のブラウザでの
自動再生許可挙動(ヘッドレス環境の制約により)は未確認 — **実機でテキストの
みの送信でも返答が自動再生されるか、引き続き確認をお願いしたい**。

## 修正: 上記の音声アンロック処理がむしろ再生を壊していた不具合(2026-07-14)

**症状**(ユーザー報告): 上記の修正後もテキスト送信時に返答が自動再生され
ず、さらに過去メッセージの「音声を再生」ボタン(手動再生)まで無反応に
なった。音声入力(STT)自体は問題なく動作し、ブラウザのタブの音声再生
インジケーター(音マーク)は反応するが、実際には何も聞こえない状態。

**原因**: `audioplayers_web`のソース(`wrapped_player.dart`)を確認したところ、
`AudioPlayer.play()`は内部で `setSource()`(新しい`<audio>`要素を作り直す
`recreateNode()`を含む)→`resume()`(共有の`AudioContext`が`suspended`なら
`resume()`を`await`)という一連の非同期処理を伴うことが分かった。前回の
実装では、無音アンロック再生(`_unlockAudioPlayback()`)を「呼びっぱなし
(fire-and-forget、`await`せず`.catchError`だけ付けて投げる)」にしていた
ため、このアンロック用の`play()`の内部処理(新しい`<audio>`要素の生成や
`AudioContext.resume()`など)がまだ完了しないうちに、後から実際の返答音声
の`play()`が同じ`AudioPlayer`インスタンスに対して呼ばれてしまうケースが
あった。この2つの`play()`呼び出しは同じインスタンスの内部状態(`<audio>`
要素、`_currentUrl`など)を同時に書き換え合う競合状態になり、タイミング
次第でアンロック用の無音クリップの`<audio>`要素が「最後に生き残る」形に
なってしまい、結果として実際の返答音声の代わりに(存在するが無音の)
アンロッククリップが再生される、という状況になっていた。これは「音声
インジケーターは反応するが何も聞こえない」というユーザー報告と正確に
一致する。手動再生ボタンについても、同じ`AudioPlayer`インスタンスを
使い回している以上、過去に投げっぱなしにしたアンロック処理がまだ内部で
残っていれば同様に巻き込まれ得るため、これも説明がつく。

**対応**:
1. `_unlockAudioPlayback()`を`await`必須の非同期メソッドに変更し、
   `startListening()`・`sendText()`の呼び出し側でも`await`するようにした。
   `play()`の実際の呼び出し自体はユーザー操作のイベントハンドラ内で(他の
   `await`より前に)同期的に発生するため、ブラウザの「ユーザー操作起点の
   再生」という判定条件は引き続き満たしつつ、`play()`の内部処理が完全に
   終わってから次の処理(実際のAPI呼び出しや、その後の返答音声の再生)に
   進むようにし、競合状態を解消した。
2. 一度アンロックに成功したら以降は繰り返さないよう、`_audioContextUnlocked`
   フラグを追加した(共有`AudioContext`は一度`resume`されればセッション中
   ずっと再生可能な状態が保たれるため、毎回無音クリップを再生し直す必要は
   ない。往復のたびに数十ms程度の遅延が発生していたのも回避できる)。

**確認**: `flutter analyze`/`flutter test`は問題なし。ヘッドレス環境の制約
により実際のブラウザでの音声再生そのものは確認できていない —
**実機で、(a) テキストのみの送信での返答自動再生、(b) 音声入力後の返答
自動再生、(c) 過去メッセージの「音声を再生」ボタンでの手動再生、の3つが
すべて問題なく音が鳴るか確認をお願いしたい**。

## 修正: 音声アンロック処理が返答音声を無音化していた不具合(真の原因)(2026-07-17)

上記2件の修正後も「読み上げ中」と表示され、ポン音も鳴り、Networkタブでも
音声ファイル自体は206で正常に取得できているのに、実際の音が一切聞こえない
という報告を受け、ユーザーにブラウザの開発者ツール(Network/Console)の
情報提供をお願いして原因を特定した。

**決め手になった情報**: Networkタブで返答の音声ファイル(`...mp3`)が
`206 media`として正常に取得されていた(=ブラウザは実際に音声再生
パイプラインを実行できていた)。かつ、アプリを介さず署名付きURLを直接
ブラウザで開いた場合は普通に音が聞こえた。「読み上げ中」は(30秒の
タイムアウトではなく)実際の音声の長さに応じて自然に終わり、ポン音も
鳴っていた。これらから、「再生自体は最後まで正常に実行されているが、
出力音量だけが0になっている」という仮説が立ち、`audioplayers`のソースを
確認して裏付けが取れた。

**原因**: `AudioPlayer.setVolume()`(および`play(..., volume: ...)`の
`volume`引数)は、その`AudioPlayer`インスタンスに**永続する設定**であり、
`play()`は`volume`引数を渡さない限り前回設定された値をそのまま使う
(`audioplayers-6.8.1`の`play()`実装: `if (volume != null) await setVolume(volume);`
— 渡さなければ何もせず、既存の値のまま)。

以前追加した音声アンロック処理で:
```dart
audioPlayer.play(AssetSource('sounds/unlock_silent.wav'), volume: 0.0)
```
としていたため、**セッション最初のこの1回の呼び出しで、共有`audioPlayer`
インスタンスの音量が0に固定されてしまっていた**。その後の実際の返答再生
(`_playAndAwaitCompletion`内の`audioPlayer.play(UrlSource(url))`)も過去
メッセージの手動再生も、同じ`audioPlayer`インスタンスを使い回しながら
`volume`を明示的に指定していなかったため、以降ずっと音量0のまま再生されて
いた。これは「アンロック」機能を実装したこと自体が、以降の全ての音声再生を
無音化するという副作用を生んでいた形になる(意図とは真逆の結果)。

**対応**:
1. アンロック用の無音再生から`volume: 0.0`指定を削除した。再生する
   WAVファイル自体がそもそも無音(全サンプル0)なので、音量を0にする
   必要は元々なかった。
2. 実際の返答/手動再生の`play()`呼び出しに`volume: 1.0`を明示的に指定
   するようにした。以前の状態(「前回設定された値に暗黙的に依存する」)は
   壊れやすい設計だったため、毎回明示的に指定することで今後同様の問題が
   起きないようにした。

**確認**: `flutter analyze`/`flutter test`は問題なし。バックエンド側
(音声ファイルの有効性・CORS設定)はcurlで直接検証済みで問題なし
(署名付きURLをダウンロードして有効なMP3であることを確認、ブラウザの
`<audio crossorigin>`と同じ形のOrigin+Rangeリクエストで`Access-Control-
Allow-Origin: *`と`206 Partial Content`が返ることを確認)。今回の原因は
完全にクライアント側(Flutterアプリの音量状態管理)にあった。

**実機での確認をお願いしたい点**: テキスト送信・音声入力どちらの場合でも
返答が自動再生されること、過去メッセージの「音声を再生」ボタンでの手動
再生も含め、実際に音が聞こえるようになったか。

## 修正: スマホで数ターン後に自動再生が止まる不具合(2026-07-17)

**症状**(ユーザー報告、スマホ実機): 最初の1〜2ターン(テキスト送信/音声
入力→手動で入力終了させて送信、のいずれも)は返答が読み上げられたが、
「無音判定による自動での入力終了」を1回経由した後、それ以降はテキスト
送信・音声入力どちらを試しても読み上げられなくなった。一貫して、過去
メッセージの「音声を再生」ボタン(手動再生)だけは常に機能した。

**原因**: モバイルのSafari(および他のモバイルブラウザ)は、一定時間
画面へのタップ操作がないと、共有の`AudioContext`を自動的に
再サスペンド(一時停止)することがある。無音判定で音声入力から自動送信
される場合、ユーザーが話してから無音判定が成立するまでの数秒間はタップ
操作が一切発生しないため、この間に`AudioContext`が再サスペンドされ
得る。一方、手動再生ボタンはタップした瞬間に`play()`が呼ばれるため、
その場でジェスチャーに紐づいた形で`AudioContext`を再開でき、常に音が
鳴っていた。

前回の修正で追加した「音声アンロック」処理(無音クリップを再生して
`AudioContext`を起こす)は、**セッション中1回だけ**しか実行しないように
していた(`_audioContextUnlocked`フラグ)。そのため、セッション開始直後の
1〜2ターンは(直前にアンロックしたばかりで`AudioContext`がまだ再開状態を
保っていたため)偶然うまくいっていたが、無音判定を待つターンで数秒間
タップが発生しない状況を経由すると`AudioContext`が再サスペンドされ、
以降は(タップの瞬間に行う手動再生を除く)自動再生が呼ばれるたびに
`audioplayers`内部で行われる自動再開の試み(`AudioContext.state ===
'suspended'`なら`resume()`する処理)がジェスチャーに紐づいていないため
効かず、無音のまま推移していた。

**対応**: `_audioContextUnlocked`フラグを削除し、**マイクをタップして
聞き取りを開始するたび**(`startListening()`)、および**送信ボタンを
押す/Enterキーで送信するたび**(`sendText()`)に、毎回この無音アンロック
再生を実行するようにした(1回きりではなく、タップ操作に紐づく形で
`AudioContext`を都度再開させる)。無音判定による自動送信自体(タイマー
発火、タップとは無関係)からの呼び出しでは根本的な解決にはならないが、
その直前にユーザーがマイクをタップした時点で`AudioContext`を再開させて
おくことで、無音判定までの数秒間の再サスペンドに対してより強くなる
狙い。

**確認**: `flutter analyze`/`flutter test`は問題なし。この対応は
ジェスチャーからの経過時間に依存するヒューリスティックな性質上、
JSレベルでの完全な保証はできない。**実機で、複数ターン連続で会話しても
(特に無音判定による自動送信を挟んでも)読み上げが継続して機能するか、
引き続き確認をお願いしたい**。もし今回の対応でも再発する場合は、発生
までの正確な操作手順(何ターン目で止まったか、直前にどの送信方法を
使ったか)を教えていただけると、次の切り分けに役立つ。

## 修正: 無音判定タイマーによる自動送信の時だけ再生されない不具合(2026-07-17)

**症状**(ユーザー報告、スマホ実機): 上記対応後、テキスト送信・音声入力を
手動で入力終了させて送信、の2パターンは安定して読み上げられるように
なった。一方で、**音声入力して無音判定タイマーの自動送信を待った場合**
だけは読み上げられなかった。かつ、無音タイマー経由で失敗した直後の
ターンでもう一度手動パターン(1・2)を試すと問題なく読み上げられた
(前回までのように以降ずっと壊れたままにはならなくなった)。

**分析**: これは前回の仮説(モバイルSafariが、タップ操作のない時間が
続くと共有`AudioContext`を自動サスペンドする)を裏付ける結果になった。
テキスト送信・手動での入力終了は、送信ボタン/マイクボタンをタップした
「直後」にAPI応答を待つだけ(応答待ち自体は無音判定のケースと同程度の
時間がかかるにも関わらず問題ない)なのに対し、無音判定を待つケースだけは
「マイクをタップしてから無音判定が成立するまでの数秒間、ユーザーが
話す・沈黙するだけでタップ操作が一切発生しない」という点で唯一異なる。
この「聞き取り中の無音時間」の間に`AudioContext`がサスペンドされ、
無音判定タイマー(`Timer.periodic`)発火による送信はジェスチャーに紐づいて
いないため、サスペンドされた`AudioContext`を再開できず、その回の返答
だけ無音になっていたと考えられる。

**対応**: 「聞き取り中(`isListening`)」の間だけ、3秒おきに無音クリップの
再生(前述の音声アンロック処理と同じもの)を繰り返す仕組みを追加した
(`_startAudioKeepAlive()`/`_stopAudioKeepAlive()`)。これにより
`AudioContext`に継続的に(無音とはいえ)アクティビティを与え続け、
無音判定が成立するまでの間サスペンドされるのを防ぐ狙い。マイクタップ時に
開始し、聞き取りが終わる全ての経路(手動での入力終了・キャンセル・
無音判定タイムアウト・音声認識エラー・ステータス変化)で確実に停止する
ようにした。

**確認**: `flutter analyze`/`flutter test`は問題なし。この対策も
ブラウザのAudioContext自動サスペンドの正確な発動条件(何秒アイドルで
サスペンドされるか等)がドキュメント化されていない前提での推測に基づく
ため、完全な保証はできない。**実機で、音声入力→無音判定タイマーによる
自動送信、のパターンでも読み上げられるようになったか確認をお願いしたい**。
なお、PC版(Windows Chrome)については今回いずれのパターンでも読み上げ
られなかったとのことだが、ユーザーからスマホ利用のみを想定しているとの
指示があったため、今回はPC版の追加調査は行っていない。

## 修正: 録音開始のラグと、入力終了を急ぐと音声が途切れる不具合(2026-07-18)

`eleven_v3`切り替え後の実機フィードバックで2件のSTT(音声入力)側の不具合が
判明した。

**録音開始のラグ**: `startListening()`内で音声アンロック処理
(`_unlockAudioPlayback()`)を`await`していたため、無音クリップの再生が
実際に始まるまで`_speech.listen()`(STT開始)が待たされていた。マイクを
タップしてから実際に聞き取りが始まるまでにラグが生じ、話し始めの数語が
拾われないことがあった。アンロック処理と実際の返答再生が競合しないよう
`await`していた経緯があったが、その競合対策は`sendText()`側で返答再生の
直前に行う`await`だけで十分に効いており(そちらは音声入力から実際の
API呼び出しまで確実に時間差があるため)、マイクタップ時のアンロックは
`await`せず投げっぱなしにしても問題ないと判断し、`await`を外した。

**入力終了ボタンを急いで押すと発話の最後が送信されない**:
`stopListeningAndSend()`が`_speech.stop()`を呼ぶ**前**に認識テキストを
読み取っていたバグ。`speech_to_text`の`stop()`は「呼ぶと最終認識結果が
非同期で1回追加で届く」仕様のため、`stop()`より前にテキストを読むと、
その最終結果が反映される前の古いテキストを送ってしまっていた。
`stop()`を呼んだ**後**に400ms待ってからテキストを読むよう順序を修正
(無音判定タイムアウト側の`_handleSilenceTimeout()`にも同じ修正を適用)。

## 修正: 上記対応後に再発した「自動読み上げされない」「ボソッと喋ると誤認識」(2026-07-18)

上記の録音ラグ修正の直後に、優先度の高い回帰が2件報告された。

**回答の自動読み上げがされなくなった**: 原因は前回のラグ修正そのものではなく、
音声アンロック処理(`_unlockAudioPlayback()`)の呼び出し元が実質3箇所
(マイクタップ時の`startListening()`、聞き取り中3秒おきのキープアライブ
タイマー、送信直前の`sendText()`)ある状態で、これらが**直列化されずに
並行実行され得た**こと。`audioplayers_web`の内部実装(`WrappedPlayer`)は
`setUrl`/`recreateNode`といった状態変更を`await`をまたいで非同期に行って
おり、同じ`AudioPlayer`に対して`play()`が重なって呼ばれると(たとえ両方が
同じ無音クリップの再生であっても)内部状態が壊れうる。3秒おきのキープ
アライブタイマーを追加したことで、この「重なり」が稀な事象から頻繁な事象に
変わり、実際の返答再生の頃には音声パイプラインが壊れている、という状態に
陥っていたと考えられる。

対応として、`_unlockAudioPlayback()`の全呼び出しを1つの実行中`Future`に
直列化した(`_unlockInFlight`)。既に実行中の呼び出しがあれば新しい
`play()`を発火せず、同じ`Future`を待つようにすることで、重複実行そのものを
構造的に防いだ。

**ボソッと喋ると誤認識される(音声入力精度の低下)**: キープアライブ
タイマーが3秒おきに(無音とはいえ)実際に音声を再生していたため、ブラウザの
エコーキャンセレーション/自動ゲイン制御(AGC)が、マイク入力に対して
「何か再生中」という前提で処理を行い、小さな声を拾いにくくしていた可能性が
高い。この仕組みが対策していた無音判定タイマーの上限は
`maxSilenceThresholdSeconds`(10秒)なので、それをカバーできる範囲で
頻度を落とし、3秒 → 10秒間隔に変更した。完全に解消しない場合は、さらに
間隔を延ばすか、別の方式(生の`AudioContext`への直接アクセスなど、現状の
`audioplayers`パッケージでは扱えない)を検討する必要がある。

## 修正: キープアライブ間隔延長後も「稀にしか自動再生されない」(2026-07-18)

上記のキープアライブ直列化+間隔延長(3秒→10秒)後も、「まれにしか
自動再生してくれない」という報告が続いた。

原因は、キープアライブタイマーの停止タイミングにあった。
`stopListeningAndSend()`/`_handleSilenceTimeout()`はいずれも
`_speech.stop()`を呼ぶ前に`_stopAudioKeepAlive()`でタイマーを止めており、
その後に呼ばれる`sendText()`のAPI待機区間(`isSending == true`、
Claude+ElevenLabsの往復で実測5秒前後かかる)は、区間開始直前の
`await _unlockAudioPlayback()`一回きりで無防備だった。この間にブラウザが
`AudioContext`を再サスペンドしてしまうと、往復完了後の返答音声の自動
`play()`はジェスチャーに紐づかない呼び出しのため、サスペンドを自力で
解除できず、自動再生が失敗する。

対応として、`sendText()`内で`isSending`区間の間だけ動く専用のキープ
アライブを追加した。この区間ではSTTは既に停止済み(マイクは閉じている)
ため、聞き取り中のキープアライブが抱えていた「頻度を上げると音声認識
精度が落ちる」というトレードオフが存在しない。そのため、聞き取り中用の
10秒間隔とは別に、より短い3秒間隔(`_sendingKeepAliveInterval`)を新設した
—往復の実測時間(~5秒)より聞き取り中の10秒間隔の方が長く、区間開始時に
タイマーを振り直しても一度もpingが発火しないまま往復が終わる可能性が
あったため。

## 修正: 自動再生に失敗すると手動再生も含めて再生が完全に止まる(2026-07-18)

上記のキープアライブ関連の修正後も、「一定時間以上かかって回答が来ると
自動再生されず、しかもその後は手動の『音声を再生』ボタンを押しても
二度と再生されない」という報告があった。

これまでの直列化(`_unlockInFlight`)は、無音アンロック再生同士の重複
呼び出しは防いでいたが、**実際の返答音声の再生(`_playAndAwaitCompletion`
内の`audioPlayer.play(UrlSource(url))`)はその直列化の対象外**だった。
そのため、`isSending`区間のキープアライブping(無音クリップの
`play()`)が実行中のまま、API往復が完了して実際の返答再生の`play()`が
呼ばれると、同じ`AudioPlayer`に対して2つの`play()`呼び出しが重なる
ケースが残っていた。以前特定した通り`audioplayers_web`の`WrappedPlayer`
は`await`をまたいで非同期に内部状態を書き換えるため、この重なりが
発生すると内部状態が壊れる。しかも壊れた状態は同じ`AudioPlayer`
インスタンスに残り続けるため、それ以降は自動再生はもちろん、同じ
インスタンスを使う手動再生(「音声を再生」ボタン)まで道連れで壊れて
いたと考えられる — これが「一回失敗するとその後ずっと再生されない」
という症状と一致する。

対応として、`audioPlayer.play()`を呼ぶ全箇所(無音アンロック再生・
実際の返答再生・手動再生)を1本の直列実行キュー(`_runOnAudioPlayer` /
`_audioOpQueue`)に統一した。呼び出し元を問わず、同じ`AudioPlayer`への
`play()`は常に前の呼び出しが完了してから次が実行されるようにすることで、
重複呼び出しによる内部状態の破損そのものを構造的に防いだ。
(`_unlockInFlight`による無音アンロック同士の重複排除は、無駄な
キューイングを避けるため引き続き残している。)

## 修正: 直列化キュー自体がハングして再生が止まる(2026-07-19)

上記の全面直列化(`_runOnAudioPlayer`)後も、「回答に時間がかかった
場合に自動再生されない」という報告があった。

原因は直列化キュー自体の設計不備だった。`_playAndAwaitCompletion`の
既存コメントに以前から明記されている通り、`audioPlayer.play()`が返す
`Future`は、ブラウザが自動再生をブロックした場合に**エラーにもならず
永遠に未解決のまま**になることがある(実際に観測済みの挙動)。
`_runOnAudioPlayer`はこの`Future`が解決するのを待ってから次のキュー
項目を実行する設計だったため、キープアライブの無音pingが1回でもこの
状態にハングすると、そのpingが未来永劫キューを塞ぎ、それ以降に
キューイングされる**全ての**`play()`呼び出し(実際の返答再生・手動再生
含む)が二度と実行されなくなっていた。さらに、`_doUnlockAudioPlayback()`
はこの同じキューを直接`await`しているため、`_unlockInFlight`も同様に
永遠に解決しなくなり、それを`sendText()`の先頭で`await`している以上、
最悪の場合は音声再生だけでなくメッセージ送信そのものまで固まりうる
状態だった。

対応として、`_runOnAudioPlayer`でキューに載せる各操作に
`_audioOpTimeout`(10秒)を設け、`action().timeout(_audioOpTimeout)`で
必ず一定時間内にキューが先に進むようにした。呼び出し元ごとに個別に
タイムアウトを設定する必要がなくなったため、`_playAndAwaitCompletion`
側で重複していた`.timeout(10秒)`は削除し、タイムアウト管理を
`_runOnAudioPlayer`に一本化した。

## 修正: ping方式のキープアライブを全面撤廃し、常時ループ再生方式へ(2026-07-19)

上記のタイムアウト修正後も、「回答生成に時間がかかると自動再生されない
(短文なら再生される)。失敗時はSafariの音声再生中マークすら出ない」
という報告があった。

これは重要な新情報だった。マークすら出ないということは、キープアライブの
無音ping自体がブロックされている(たまに失敗しているのではなく、ある
時間を超えると一貫してブロックされる)ことを意味する。つまりブラウザの
自動再生ポリシーは「最近音声が再生されていたか」ではなく、「実際の
ユーザー操作からどれだけ時間が経過したか」で判定している可能性が高く、
その場合はJSの`Timer`から新規に`play()`を呼び直しても「ユーザー操作」
とはみなされないため、pingの頻度をいくら上げても原理的に解決しない。

対応として、キープアライブの方式そのものを変更した。数秒おきに新しい
`play()`呼び出しを発火する(discreteな「ping」)方式をやめ、ユーザー操作
(マイクタップ・送信操作)が得られた瞬間に無音クリップを**ループ再生**で
1回だけ開始し、以後は新たな`play()`呼び出しを一切発行しない方式にした。
一度開始されたループ再生は、ブラウザにとっては「継続中の同一再生
セッション」であり、新規の自動再生要求とはみなされないため、都度の許可
判定を受けずに再生され続ける(chatアプリの通知音などで広く使われている
既知の手法)。実際の返答が届いたら、同じ`AudioPlayer`のソースを返答音声に
差し替え(ループを中断)、再生完了後にループ再生を再開して次の(長くなる
かもしれない)待ち時間に備える。

この変更により、数秒おきにpingを打つ仕組み(`_startAudioKeepAlive`/
`_stopAudioKeepAlive`/`_audioKeepAliveTimer`とその間隔調整)は不要になり
丸ごと削除した。副次的に、「短い間隔でpingを打つとエコーキャンセレー
ション/AGCが音声入力精度を落とす」というこれまで抱えていたトレードオフ
(短くしたいが認識精度が落ちる)自体もなくなった — 継続的な無音ループは
新規の「再生開始」イベントを繰り返し発生させないため。

## 修正: 送信ボタンの二重連打で同じ内容が2回送信される(2026-07-19)

`sendText()`は先頭で`isSending`をチェックしていたが、実際に`isSending = true`
を代入するのは`await`(当時は無音アンロック再生の待機)より**後**だった。
Dartの非同期関数は最初の`await`まで同期的に実行されるため、この
チェックと代入の間に隙間があると、素早い二重タップの両方がこの隙間を
すり抜けてチェックを通過し、同じ内容を2回送信してしまう。

対応として、`isSending = true`の代入(および入力欄のクリア・
`notifyListeners()`)を関数の先頭、最初の`await`より前の同期処理として
まとめて実行するよう並び替えた。これによりチェックと代入が実質的に
アトミックになり、二重タップの2回目は必ず`isSending`が`true`になった
後に評価されて弾かれるようになった。

短いフィラーフレーズをあらかじめ1回だけElevenLabsで合成し、アプリの
静的アセットとして同梱**する方式にした(`assets/sounds/thinking_1.mp3`
〜`thinking_3.mp3`、`えっとね…ちょっと考えるね。`ほか、system-prompt.md
のフィラー語彙に合わせた3種)。この方式なら追加のClaude/ElevenLabsの
トークン・クレジット消費は実質ゼロ(初回合成のみ)で、往復のたびにレイ
テンシが増えることもない。

`sendText()`でAPI呼び出しを開始した後、`_thinkingFillerDelay`(3秒)
以内に応答が返らなければ、3種類からランダムに1つを選んで再生する
(`_playThinkingFiller()`)。実際の返答が届いたら、フィラー再生が完了
するまで待ってから(`_fillerPlaybackDone`)本編を再生し、重ならないように
した。フィラー再生も、無音ループ機構(`_ensureSilentLoopPlaying`)や
直列化キュー(`_runOnAudioPlayer`)と同じ`audioPlayer`インスタンスを
経由させている — 専用の別プレイヤーを使うと、そのプレイヤー自身に
改めて「ユーザー操作による解放」が必要になってしまうため、既にループ
再生でアンロック済みの共有プレイヤーを使う方が確実。

## 修正: テキスト入力中にEnterで送信されてしまい改行できない(2026-07-19)

`TextField`に`textInputAction: TextInputAction.send`と`onSubmitted`を
設定していたため、複数行入力を意図した`minLines`/`maxLines`の設定に
関わらず、Enterキーが常に送信として扱われていた。`textInputAction`を
`TextInputAction.newline`に変更し、`keyboardType: TextInputType.multiline`
を明示、`onSubmitted`ハンドラを削除した。これによりEnterは常に改行として
扱われ、送信は送信ボタンのみで行われるようになった。

## 対応: 音声入力が入らない/送信されない疑いの調査(2026-07-19)

`sendText()`・`stopListeningAndSend()`・`_handleSilenceTimeout()`・
`startListening()`まわりのコードを読み返したが、直近の変更(直列化
キュー・二重送信対策・考え中フィラー)が音声認識や送信のロジックパス
自体を壊している箇所は見つからなかった(いずれも音声再生に関する変更で、
STTの開始・結果反映・送信のフローは変更していない)。

一方で、有力な仮説が1つある。前回の修正で「無音クリップを常時ループ
再生し続ける」方式に切り替えたが、これは以前の「短いpingを数秒おきに
鳴らす」方式より**再生が常時アクティブ**になる。以前すでに「短いpingが
ブラウザのエコーキャンセレーション/自動ゲイン制御(AGC)に干渉して、
小さな声が誤認識される」という実績のある不具合パターンがあり、常時
ループ化でこの干渉がむしろ悪化した可能性がある。

対応として、ループ再生の`volume`を明示的に`0.0`に設定した。以前は
「`volume`はAudioPlayerインスタンスに永続し、明示的に指定しない限り
次の再生に引き継がれる」ため意図的に指定していなかったが、現在は
実際の返答再生・フィラー再生のいずれも呼び出し側で`volume: 1.0`を
明示するようになっている(直列化まわりの一連の修正で追加済み)ため、
ループ側だけ`0.0`にしても安全になった。ゲインをゼロにすることで、
ブラウザ側が「何か出力中」と判定する材料を減らせる可能性がある。

ただし、これは推測に基づく防御的な対応であり、実際の原因かどうかは
実機で確認できていない。「音声入力が入らない」の具体的な症状
(「音声を認識しています」の表示は出るか、話している間に文字が入力欄に
出るか、無音判定タイムアウト後に送信されないのか、など)が分かれば、
より的確に切り分けられる。

## 修正: 2回目以降テキスト入力ができない(2026-07-19)

送信のたびに`TextField`の`enabled: !_controller.isSending`をfalse→true
と切り替えていたことが原因と考えられる。Flutter Webのテキスト入力
プラグインには、`TextField.enabled`をfalseからtrueに戻しても内部の
DOM要素側のフォーカス可能性が完全には復元されないという既知の問題が
あり、これが「1通目は打てるが2通目以降は入力を受け付けない」という
症状と一致する。

対応として、`enabled`の切り替えそのものを撤廃した。送信中の二重送信
防止は元々`sendText()`内部のガード(`isSending`チェック)と送信ボタン側の
`controlsLocked`で担保されており、テキスト欄自体を無効化する必要は
機能的にはなかった。`isPlayingReply`中にテキスト欄を有効のままにして
「先読み入力」を許している既存方針と合わせ、`isSending`中も常時
入力可能にすることで、この切り替え自体をなくしトグルに起因する不具合を
根本から回避した。

あわせて、`_playThinkingFiller()`内の`completer`が、フィラー再生が
実際には始まらなかった場合(`_runOnAudioPlayer`のタイムアウトなど)に
一切completeされないまま残るバグを発見・修正した。`sendText()`側は
`_fillerPlaybackDone`(この`completer.future`)を返答再生前に`await`して
いるため、直らないとそのターンの自動再生が永久にハングしたまま止まる
状態になり得た。`finally`ブロックで`completer`を確実にcompleteするよう
修正。

## 修正: 無音自動送信タイマーを廃止し、明示タップのみの送信に統一(2026-07-22)

**症状**(ユーザー報告、iPhone Safari実機、スクリーンショット添付): 返答の
テキストは表示されるが自動再生されず、しばらくしてから(何も再生していない
のに)ブラウザのタブの音声再生インジケーターだけがONに戻る、という現象が
「長考」(応答生成に時間がかかる)時に起きていた。ユーザー自身の分析による
と、無音判定タイマーによる自動送信の場合にほぼ100%発生していた。

**対応1: 無音自動送信タイマーの完全廃止**。マイク入力は常に明示的操作のみ
(タップで開始→もう一度タップで終了・送信、長押しでキャンセル)に統一した。
`ConversationController`から`_silenceTimer`/`silenceCountdown`/
`silenceThresholdSeconds`/`_restartSilenceCountdown()`/
`_handleSilenceTimeout()`一式を削除し、`chat_screen.dart`の無音判定時間
設定メニュー・カウントダウン表示も削除した。

**対応2(根本原因): `audioplayers`パッケージへの依存を廃し、Web Audio APIへ
直接書き換え**。シークレットモードで検証したところ無音自動送信は解消されて
いたが、「長考時に自動再生されない」自体は解消していなかった。原因は
`audioplayers_web`の`WrappedPlayer.setUrl()`実装にあった
(`~/.pub-cache/hosted/pub.dev/audioplayers_web-5.3.0/lib/wrapped_player.dart`):
再生ソースのURLを変える(=無音アンロッククリップ→実際の返答音声への
差し替え)たびに、内部で`release()`(既存の`<audio>`要素を破棄)→
`recreateNode()`(新しい`<audio>`要素を生成し直す)を毎回行っている。
これまでの一連の「無音クリップをループ再生してブラウザの自動再生ロックを
解除しておく」対策は、**同じ`<audio>`要素が継続再生され続けることを前提**
にしていたが、実際には差し替えのたびに新しい要素が作られ直しており、
iOS Safariはその新しい要素に対して(たとえ同じ`AudioContext`に接続されて
いても)改めて「実際のユーザー操作に紐づく再生」を要求する。これが
「アンロックできているように見えるのに、いざ返答音声に差し替えた瞬間に
再生が止まる」というユーザー報告(音声マークが返答のタイミングでOFFに
なり、その後アンロックループの再起動でONに戻る)と正確に一致する。

対応として、`lib/services/audio_playback_service.dart`を`audioplayers`
パッケージを一切使わない実装に書き換えた。単一の`AudioContext`をアプリ
起動後の最初のユーザー操作(マイクタップ/送信タップ)で1回だけ`resume()`
し(以後はセッション中ずっと再開状態を保つ)、返答音声は`fetch()`→
`decodeAudioData()`→`AudioBufferSourceNode`経由で同じ`AudioContext`に
対してその都度新しいノードとして再生する。Web Audio APIの
`AudioContext`は、一度ユーザー操作で`resume()`されれば、その後どれだけ
時間が経ってから新しいバッファを再生しても改めてユーザー操作を要求され
ない(`<audio>`要素の再生成とは異なる)ため、"長考"による遅延の影響を
受けなくなる。`pubspec.yaml`から`audioplayers`一式を削除し、代わりに
(既に間接依存として解決されていた)`web`パッケージを直接の依存に追加した。

**副作用として、このアプリは今後Web専用になった**: `package:web`は
`dart:js_interop`ベースのWeb専用APIであり、Dart VM(`flutter test`の
デフォルトターゲット)やネイティブAndroid/iOSではコンパイルできない。
`flutter test`はデフォルトでVM向けにコンパイルしようとして失敗するため、
`test/widget_test.dart`に`@TestOn('chrome')`を付与し、**`flutter test`は
必ず`flutter test --platform chrome`(または`CHROME_EXECUTABLE`を設定した
上でのデフォルト実行)で行う**必要がある(`@TestOn`が無い場合、VM向け
コンパイルがpackage:web絡みでエラーになる。付けておけば`--platform`を
指定し忘れても「No tests ran」で静かにスキップされるだけで済む)。
将来ネイティブモバイル版に展開する際は、この`AudioPlaybackService`だけ
プラットフォーム別実装(条件付きimportで`dart:io`環境向けの別実装、例:
`audioplayers`や`just_audio`に戻す等)に分岐させる必要がある。

**確認**: `flutter analyze`/`flutter test --platform chrome`は問題なし。
`AudioContext.resume()`および実再生の完了待ちには、ジェスチャーなしの
環境(自動テスト等)で無期限にpendingし得ることが判明したため、両方に
タイムアウト(それぞれ5秒・30秒、失敗時は握りつぶして処理を続行)を設けて
ある。本番環境(実AWS)へビルド・デプロイ済み。**実機(iPhone Safari)で、
特にテキスト送信直後や応答が数秒以上かかる場合の自動再生が安定して
機能するようになったか、引き続き確認をお願いしたい**。

## 修正: Web Audio API化の副作用2件(ミュートスイッチ無視・再生の巻き戻り)(2026-07-22)

上記のWeb Audio API化の直後にレビューがあり、以下2つの新たな不具合が
報告された:「数ラリーは正常だったが、長文回答の後で自動再生も手動再生も
一切できなくなり、しばらくしてブラウザに戻ると、鳴らなかった分がまとめて
同時に再生された」「本体の音量を下げても音量が変わらず、リフレッシュ後は
サイレントモード(マナーモード)でもそこそこの音量で鳴ってしまった」。

**不具合A: ハードウェアのミュートスイッチ・音量ボタンを無視して再生される**。
iOS Safariは、ページ内で通常の`<audio>`/`<video>`要素(`HTMLMediaElement`)が
一切再生されていない場合、そのページのオーディオセッションを「playback」
カテゴリ(サイレントスイッチを無視し、音量ボタンの挙動も不安定になりうる)
として扱う。前回の書き換えで`audioplayers`(`<audio>`要素ベース)を完全に
撤去し生の`AudioContext`のみにしたことで、この条件に該当してしまっていた。

**対応**: `unlock()`実行時に、無音WAVアセットから生成したBlob URLを`src`に
持つ、通常の(Web Audio APIを経由しない)`<audio>`要素を1つだけ生成し、
セッション中ずっとループ再生し続けるようにした(`_ensureAmbientCategoryLoop`)。
この要素の`src`はセッション中一切変更しない(常に同じBlob URL)ため、
前回撤去した「`src`差し替えのたびに新しい要素が生成され、都度ジェスチャーを
要求される」問題は再発しない。この要素が存在し続けることで、ページ全体の
オーディオセッションが「ambient」カテゴリ(ミュートスイッチ・音量ボタンに
正しく従う)に保たれ、別経路で鳴らしている`AudioContext`側の再生にも
この設定が適用される。

**不具合B: 長考後に再生が完全に止まり、後でまとめて再生される**。
`AudioContext`はOS側の都合(画面ロック、バックグラウンド化、電話の着信など)
で、アプリのコードとは無関係にサスペンド(一時停止)されることがある。
`AudioBufferSourceNode.start()`は`AudioContext`の内部クロックを基準に再生を
予約するだけなので、そのクロックがサスペンドで止まっていると、ノードは
「開始はした」が実際には何も鳴らないまま待機し続け、その後コンテキストが
再開した瞬間に、それまで溜まっていたノードが一斉に鳴ってしまう。前回の
実装は、`play()`呼び出し時にコンテキストの状態を確認せず、完了待ちが
タイムアウト(30秒)した後もノード自体を停止・破棄していなかったため、
この「取りこぼされたノードがゾンビとして残り続ける」状態が起きていた。
自動再生・手動再生いずれの経路もこの問題を等しく踏むため、「自動再生も
手動再生も両方止まる」というユーザー報告と一致する。

**対応**: `play()`の冒頭で、実際に音声データをスケジュールする直前に改めて
`resume()`を試み、それでも`state !== 'running'`であれば**その場で例外を
投げて諦める**(=将来いつか再開した時に鳴るという状態を許さない)ように
した。あわせて、新しい`play()`を開始する前・完了待ちタイムアウト時・
`stop()`呼び出し時のいずれでも、直前のノードを必ず`stop()`+`disconnect()`
してから次に進むようにし(`_killActiveSource()`)、生きているノードが同時に
2つ以上存在しない状態を構造的に保証した。

**確認**: `flutter analyze`/`flutter test --platform chrome`は問題なし。
本番環境へビルド・デプロイ済み。ヘッドレス環境の制約上、iOSのミュート
スイッチ・OSレベルのサスペンド/再開そのものは実機でしか検証できない。
**実機で、(a) 本体をサイレントモード/音量ゼロにした状態で音声が鳴らなく
なっているか、(b) 長文回答や複数ラリーを経ても自動・手動再生が安定して
機能し続けるか、引き続き確認をお願いしたい**。

## 再設計: `AudioPlaybackService`をWeb Audio API単独から`<audio>`要素併用へ(2026-07-22)

v0.2の対応後も「本体の音量を下げても音声ボリュームが変わらない」「サイレント
モードでもそこそこの音量で鳴る」が実機で再解決していない、との報告を受けた。
今回は既存の実装に固執せず、根本から調査・再設計した。

### 調査で判明したこと

外部資料(Apple Developer Forums、および複数の実装記事)で以下が確認できた:

- **iOS Safariのミュート(サイレント)スイッチ・ハードウェア音量ボタンは、
  実際の`<audio>`/`<video>`要素(`HTMLMediaElement`)経由の再生にしか
  効かない。Web Audio API(`AudioContext`/`GainNode`)経由の出力には
  一切効かない**(`GainNode.gain`はグラフ内の相対的な音量調整であり、
  ハードウェア側のミュート/音量状態を復元する仕組みではない)。前回・
  前々回の対応(無音`<audio>`要素を"ambient"カテゴリ確保のためだけに
  裏で鳴らしつつ、実際の返答再生自体はWeb Audio APIの`GainNode`経由)は、
  この核心を見誤っていた ——「ページ全体がambientカテゴリになる」ことと
  「Web Audio API自体の出力がミュート/音量に従う」ことは別問題であり、
  後者は`<audio>`要素を経由しない限り原理的に解決しない。
- 一方で、**同一の`<audio>`要素を(生成し直さずに)使い回す限り、一度
  実際のユーザー操作で再生できれば、以降は`src`を差し替えて`play()`し
  直すだけで、新たなユーザー操作なしに再生できる**、というのがiOSでの
  標準的な「アンロック」パターンだと確認できた。これまでの一連の不具合
  (`audioplayers`の`WrappedPlayer`が`src`変更のたびに要素を破棄・再生成
  していた)は、「`<audio>`要素方式そのもの」が悪かったのではなく、
  「要素を作り直していたこと」が原因だったと判明した。
- 副次情報として、モバイルSafariでは`HTMLMediaElement.volume`が読み取り
  専用(設定しても無視される)であることも確認した。相対音量の調整が
  必要な場面では`.volume`に頼らない設計にする必要がある。

### 新しい設計

`lib/services/audio_playback_service.dart`を全面的に書き直した:

- **返答音声(実際にユーザーが聞く声)**: セッション開始後、生成したまま
  一度も破棄しない単一の`<audio>`要素(`_replyElement`)で再生する。
  最初のユーザー操作(マイク/送信タップ)で無音アセットを一度再生して
  アンロックし、以降は同じ要素の`src`を返答の音声URLに差し替えて
  `play()`するだけ。これにより「長考でも自動再生が途切れない」(前回の
  Web Audio API化の目的)と「ミュートスイッチ・音量ボタンに正しく従う」
  (今回の再設計の目的)の両方を同時に満たす。要素を絶対に作り直さない
  ため、`audioplayers`が抱えていた問題は再発しない。
- **「ポン」の合図音**: あくまで補助的なUIフィードバックであり、ユーザーが
  実際に聞きたい音声コンテンツではないため、従来通りWeb Audio APIの
  `AudioBufferSourceNode`+`GainNode`(音量0.18)で鳴らす方針を維持した
  (多くのアプリがUI効果音についてはミュートスイッチを無視する設計を
  許容しているのと同じ判断)。
- 返答再生の直前に`<audio>.play()`が失敗(reject)した場合は即座に例外を
  投げる。Web Audio API版で起きていた「サスペンドされたまま`start()`が
  静かに予約され、後でまとめて鳴る」という問題は、`<audio>`要素の
  `play()`Promiseがブロック時に正しくrejectすることに依存する形になり、
  構造的に起きなくなった(念のため完了待ちタイムアウトは維持)。

### 確認

`flutter analyze`/`flutter test --platform chrome`は問題なし。ビルド後の
アセットパス(`assets/assets/sounds/unlock_silent.wav`)を実際のビルド
成果物で確認し、`unlock()`が同期的に(アセットの非同期読み込みを挟まずに)
`<audio>.src`を設定できることを保証した。ヘッドレス環境+プロキシ制約の
組み合わせにより、この環境では実際の会話フローをブラウザ上で最後まで
自動検証することはできなかった(CanvasKitはローカル同梱ビルドで解決できた
ものの、会話APIへの実際のネットワーク到達性がこの検証環境固有の制約で
不安定だった)。**実機(iPhone Safari)で、(a) サイレントモード/音量調整に
再生音量が正しく追従するか、(b) 長文回答や複数ラリーを経ても自動・手動
再生が安定して機能し続けるか、引き続き確認をお願いしたい**。

## 今後(モバイル版への展開時)

- `flutter create` の標準構成のため `flutter build ios` / `flutter build apk` は
  そのまま試せるはず(未検証)。ただし上記の通り`AudioPlaybackService`は
  現在Web専用実装のため、モバイル版に進む際はプラットフォーム別実装への
  分岐が必要
- モバイルではシークレットのビルド埋め込みはさらにリスクが高い
  (アプリのバイナリを解析されると同様に漏れる)ため、上記の見直しは
  モバイル版に進む前に必須
- 「送信しただけでは自動再生されず、手動の『音声を再生』ボタンが必要」な
  現象(ブラウザの自動再生ポリシー)は**Web固有の制約**。ネイティブアプリ
  (iOS/Android)にはこの制約が無いため、モバイル版では送信だけで自動再生
  される想定
